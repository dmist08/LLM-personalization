## Why the Fine-Tuned Model for Extraction

The style vector is computed as:

```
v_u = mean(h_pos - h_neg)

where:
h_pos = hidden state when model sees (article + real_headline)
h_neg = hidden state when model sees (article + agnostic_headline)
```

The agnostic headline in `h_neg` was already generated by Gemini in Phase 2. It's just a string sitting in your CSV. You're not running Gemini again during extraction.

The question is: which model's hidden states do you extract from?

If you use the **base model**, its internal representations don't know anything about individual journalist styles. The difference `h_pos - h_neg` captures some signal but it's noisy — the model wasn't trained to care about user identities.

If you use the **fine-tuned model**, it has learned to associate user IDs with stylistic patterns during training. Its internal representations are organized around style. The difference `h_pos - h_neg` captures a much cleaner style signal because the model already knows user_1042 writes differently from user_0871.

This is the whole point of doing Phase 3 before Phase 4.

## Can We Use LLaMA-3.1-8B as Mg?

Yes, and it's actually a good idea — with one important condition.

You must use the **base model** (pre-fine-tuning), not the fine-tuned model. Here's why:

After fine-tuning, your model has learned style associations — it knows user_1042 writes a certain way. If you use the fine-tuned model to generate agnostic headlines, those headlines may carry residual style from training. The "agnostic" baseline is no longer truly style-free, and your contrastive difference `h_pos - h_neg` is polluted.

The base model has no style associations at all — it will generate the same generic headline regardless of which journalist's article you feed it. That's exactly what you want for the negative side of the contrastive pair.

**Practical setup:**

```python
# Load BASE model (no fine-tuning) for agnostic headline generation
# This runs locally on Lightning AI — no API needed, no rate limits
from transformers import AutoModelForCausalLM, AutoTokenizer, pipeline
import torch

base_model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3.1-8B-Instruct",
    torch_dtype=torch.float16,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained(
    "meta-llama/Meta-Llama-3.1-8B-Instruct"
)

pipe = pipeline(
    "text-generation",
    model=base_model,
    tokenizer=tokenizer,
)

def generate_agnostic_headline(article_body):
    prompt = f"""<|begin_of_text|><|start_header_id|>user<|end_header_id|>
Generate a neutral, factual headline for this article. Be direct, no style.

Article: {article_body[:1500]}<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>
Headline:"""

    out = pipe(prompt, max_new_tokens=50, temperature=0.1, do_sample=False)
    generated = out[0]["generated_text"].split("Headline:")[-1].strip()
    return generated.split("\n")[0].strip()
```

**The advantage over Groq:** No API rate limits. You run it on Lightning AI L4, generate all 7,000 agnostic headlines in one batch job in ~2 hours, save to CSV, done. No waiting across multiple days.

**The disadvantage:** Costs Lightning AI compute time (~2h on L4 = ~2.4 credits). If you want to save credits, use Groq for free and spread it over one day.
